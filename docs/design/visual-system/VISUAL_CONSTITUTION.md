# Visual Constitution

## Purpose

This document defines the non-negotiable visual identity of Athanor.

It exists so the dashboard stops drifting between partial styles and becomes one coherent operator system.

## Plain-language identity

Athanor should feel like:

- a premium control room
- a dense but calm operational console
- a trustworthy system that can stay open for hours
- a modern, technical environment with clear hierarchy
- a dark interface with strong depth, not a black void
- expressive when necessary, but never theatrical for its own sake

Athanor should not feel like:

- a furnace
- an alchemical chamber
- a mystical product
- a rustic or earthy dashboard
- a generic SaaS admin panel
- a game HUD
- a neon cyberpunk demo

## Product-success lens

The visual system passes only if it makes these easier:

- understanding what the system is doing
- telling agents what to work on
- seeing whether work is blocked, waiting, degraded, or done
- knowing what needs approval right now
- understanding why a cloud or sovereign lane was chosen
- recovering from failure without guesswork

This is not a branding exercise detached from control. It is an operator-effectiveness layer.

## Selected family

The locked implementation family is:

`Carbon Ops`

That means:

- deep black, charcoal, and neutral-gray surfaces
- bright neutral highlights and governed signal accents only where they carry meaning
- no page-wide blue wash or atmospheric color fog
- matte industrial materials over glossy sci-fi styling

## Brand metaphor shift

The old furnace/alchemical metaphor remains historically important to the product name, but it is no longer the active visual language for the dashboard.

The active visual metaphor is:

`futurist control room`

That means:

- mission-control clarity
- composable operational surfaces
- disciplined signals
- layered materials
- consistent telemetry grammar

It does not mean:

- NASA cosplay
- excessive space iconography
- green CRT nostalgia
- cinematic chrome overload

## Visual principles

### 1. Signal outranks decoration

If an element is colored, animated, or elevated, it must convey a real difference in:

- severity
- state
- domain
- focus
- action priority

### 2. Depth comes from layered materials, not random shadows

The interface should create hierarchy through:

- surface tiers
- border and edge-light discipline
- atmospheric background treatment
- restrained shadowing

Not through:

- heavy glow
- muddy gradients
- ornamental highlights

### 3. Multi-signal does not mean multicolor chaos

The system supports multiple signal families, but they must be governed.

At any moment, a surface should clearly distinguish:

- structure
- domain
- severity
- interaction state

Only one of those may dominate visually.

### 4. Domain color never beats severity color

If a domain accent and a severity state collide, severity wins.

Examples:

- a monitoring panel may use a cyan domain accent in nominal conditions
- if that same panel is degraded, the degraded state must visually override the domain accent

### 5. Charts are system instruments, not decoration

Charts should read like instruments:

- legible quickly
- consistent across routes
- color-stable
- direct-labeled whenever possible

### 6. Typography carries density

The interface should remain information-dense.

Density should be made readable through:

- role-based type
- spacing rhythm
- strong scale decisions
- disciplined label treatment

Not through deleting information.

### 7. Motion is for comprehension

Motion should help with:

- state change
- focus shift
- escalation
- layout continuity

It should not exist just to make the UI feel "alive."

### 8. Mobile is first-class, not compressed desktop

The shell, cards, charts, and controls must feel intentional on both:

- desktop
- mobile PWA

## Non-negotiable visual rules

### Color

- No core brand reliance on brown, bronze, sepia, ember, or furnace-like glow
- No warm metallic framing as the primary identity
- No page-level atmospheric blue haze
- No route-level accent invention outside the approved multi-signal grammar

### Surfaces

- No flat black backgrounds
- No all-card surfaces using nearly identical fills and borders
- No one-size-fits-all panel styling across shell, chrome, drawers, metrics, and overlays

### Typography

- No literary or ornamental headline tone
- No inconsistent heading voice between routes
- No treating headings, labels, and metrics as the same typographic family and weight

### Motion

- No continuous ambient animation that competes with reading
- No pulsing or glowing on nominal states unless explicitly part of the signal grammar

### Charts

- No ad hoc chart colors per route
- No using warning/danger colors as category colors unless the data actually encodes warning/danger

## Final visual test

If a new screenshot is shown without context, it should immediately read as:

- a serious operator surface
- premium and modern
- charcoal-neutral and signal-driven
- one product, not a set of unrelated pages

If it reads as earthy, mystical, rustic, or gamey, it fails.
