## Recovery Principles

- Pause automation first. Recovery is an operator posture change before it is a service restart sequence.
- Restore stateful dependencies before control-plane surfaces and control-plane surfaces before product surfaces.
- Use registry truth for current host placement and service URLs.
- Do not reintroduce secrets through tracked docs while restoring a host.
- Every drill or real recovery should produce evidence: what was restored, how it was verified, and what still needs cleanup.

## Recovery Order

### Phase 0: Stabilize posture

1. Use the governor/operator surface to pause affected automation lanes.
2. Capture the current incident state before restarting anything:
   - failing services
   - latest backup timestamps
   - current registry version
   - operator-visible impact
3. Confirm whether the operation is:
   - non-destructive rehearsal
   - destructive restore
   - host rebuild

### Phase 1: Restore shared state on `vault`

Suggested restore order for this snapshot:

1. `redis`
2. `qdrant`
3. `neo4j` / `neo4j_http`
4. any other shared data store required by the incident scope

The success condition for this phase is not “container started”; it is “data is readable and the next layer can reconnect without schema or auth surprises.”

### Phase 2: Restore routing and observability

Bring back:

1. `litellm`
2. `prometheus`
3. `grafana`
4. `langfuse`
5. `ntfy`
6. `uptime_kuma`

These services make later phases observable and keep the operator from working blind.

### Phase 3: Restore core control plane

Bring back, in dependency order:

1. `agent_server` on `foundry`
2. `gpu_orchestrator` on `foundry`
3. `ws_pty_bridge` on `workshop`
4. `quality_gate` on `dev`
5. `dashboard` on `dev`

If a control-plane service comes back without auth or without its stateful dependencies, it is not recovered yet.

### Phase 4: Restore supporting data and product surfaces

Bring back remaining supporting services only after the core path is healthy:

- `embedding`
- `reranker`
- `semantic_router`
- `gateway`
- `memory`
- `comfyui`
- `speaches`
- `miniflux`
- `stash`
- scaffold services when needed

## Recovery Evidence

Each monthly drill or real recovery should record:

- store or service tested
- backup source selected
- rehearsal vs destructive mode
- commands executed
- validation performed
- result
- rollback or reconnect notes
- follow-up gaps

Generated artifact paths for the current non-destructive loop:

- `audit/recovery/restore-drill-latest.json`
- `audit/automation/contract-healer-latest.json`

These are generated by:

- `python scripts/generate_recovery_evidence.py`
- `python scripts/run_contract_healer.py`

Until the evidence path is fully generated, treat missing evidence as an incomplete drill.

## Restore Validation Checklist

After each phase:

1. Verify health endpoints or service-specific read checks.
2. Verify auth still works from the intended client, not just localhost.
3. Verify dependent services can reconnect.
4. Verify the operator surface reports the restored component accurately.
5. Record the result before moving to the next phase.
